# Mutation Testing

A test suite that passes 100% of tests with 30% code coverage tells
you almost nothing. Mutation testing answers the real question: if a
bug is introduced into this code, will the tests catch it?

## How Mutation Testing Works

1. The mutation tool takes the source code and creates hundreds of
   "mutants" — copies of the code with deliberate small faults injected
2. Each mutant is run against the full test suite
3. If the test suite catches the mutation (a test fails): the mutant
   is **killed** ✓
4. If the test suite misses the mutation (all tests pass): the mutant
   **survives** ✗
5. Mutation score = killed mutants / total mutants × 100%

A surviving mutant means: if this specific bug existed in production,
your tests would not catch it.

---

## Common Mutation Operators

Stryker applies these mutations automatically:

| Mutation | Example | What it tests |
|---|---|---|
| Arithmetic operator | `+` → `-` | Are arithmetic results verified? |
| Logical operator | `&&` → `\|\|` | Are boolean conditions tested for both values? |
| Comparison | `>` → `>=` | Are boundary conditions tested? |
| String literal | `"active"` → `""` | Are string values verified? |
| Block removal | Remove function body | Is the function's output verified? |
| Conditional negation | `if (x)` → `if (!x)` | Are both branches tested? |
| Array declaration | `[1,2,3]` → `[]` | Are array contents verified? |

## Stryker Configuration

```bash
pnpm add -D @stryker-mutator/core @stryker-mutator/vitest-runner \
  @stryker-mutator/typescript-checker
```

```typescript
// stryker.config.mts
import type { Config } from '@stryker-mutator/core';

export default {
  testRunner: 'vitest',
  checkers: ['typescript'],
  tsconfigFile: 'tsconfig.json',
  mutate: [
    'src/**/*.ts',
    'src/**/*.tsx',
    '!src/**/*.test.ts',
    '!src/**/*.spec.ts',
    '!src/**/*.d.ts',
  ],
  reporters: ['html', 'progress'],
  htmlReporter: { fileName: 'reports/mutation/index.html' },
  thresholds: {
    high: 80,    // green — good mutation coverage
    low: 60,     // yellow — needs improvement
    break: 50,   // red — pipeline fails below this score
  },
  timeoutMS: 30000,
  concurrency: 4, // run 4 mutants in parallel
} satisfies Config;
```

```bash
# Run mutation tests
pnpm stryker run

# Run on a specific file only (faster feedback during development)
pnpm stryker run --mutate "src/features/payment/payment.service.ts"
```

## Interpreting Mutation Results

**High-value surviving mutants to investigate:**

```typescript
// If this mutant survives:
// Original:  if (balance >= amount)
// Mutant:    if (balance > amount)
// → Your tests don't verify the boundary case where balance === amount
// → Add: expect(transfer(100, 100)).resolves.not.toThrow()

// If this mutant survives:
// Original:  await auditLog.record(event)
// Mutant:    (function body removed)
// → Your tests don't verify that audit logging actually happens
// → Add: expect(auditLog.record).toHaveBeenCalledWith(expect.objectContaining({...}))
```

**Surviving mutants that are acceptable to ignore:**

```typescript
// Equivalent mutants — different code, same behavior
// Original:  return items.filter(i => i.active === true)
// Mutant:    return items.filter(i => i.active !== false)
// → Functionally equivalent — safe to ignore with a comment

// Trivial mutants in logging/debugging code
// Original:  console.error('Failed:', error.message)
// Mutant:    console.error('', error.message)
// → Not worth testing string literal content in log messages
```

## Mutation Score Targets

| Context | Target mutation score |
|---|---|
| Core business logic (payments, auth, data integrity) | ≥ 85% |
| Service layer (API handlers, orchestration) | ≥ 75% |
| Utility functions | ≥ 70% |
| UI components | ≥ 60% |
| Configuration and bootstrapping | Not required |

**For AI-generated code specifically:** Target ≥ 80% for any
module generated by an AI agent. AI code is more likely to contain
subtle logical errors that only mutation testing reliably catches.

## Integrating Mutation Tests into CI

Mutation tests are slow — running the full suite on every PR is
impractical. The recommended cadence:

```yaml
# Run mutation tests on schedule, not on every PR
# .github/workflows/mutation.yml
on:
  schedule:
    - cron: '0 2 * * *'  # nightly at 2 AM
  workflow_dispatch:      # or manually triggered

jobs:
  mutation:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Stryker
        run: pnpm stryker run
      - name: Upload mutation report
        uses: actions/upload-artifact@v4
        with:
          name: mutation-report
          path: reports/mutation/
```

Run mutation tests on PR only for files changed in the PR:
```bash
# Get changed files
CHANGED=$(git diff --name-only origin/master...HEAD | grep "\.ts$" | grep -v "\.test\.")
pnpm stryker run --mutate "$CHANGED"
```
